Accessibility Metadata Format

W3C Community Group Draft

Draft publication date: 2025-11-28

This specification defines a machine-readable metadata format for accessibility evaluations of web projects, themes, templates, components, applications, and other digital products. The format is intended to support interoperability between tools, simplify the creation of accessibility statements, and enable automated integration into content management systems and continuous integration workflows.

1. Introduction

Existing W3C technologies such as EARL (Evaluation and Report Language) and ACT Rules provide mechanisms for expressing accessibility test results and test procedures. However, they do not define a project-level metadata structure for describing:

  • project identity and versioning
  • legal basis and jurisdiction
  • audits and conformance summaries
  • known issues
  • planned improvements
  • contact information
  • links to accessibility statements

The Accessibility Metadata Format fills this gap by defining a JSON vocabulary that represents the final evaluation state of a digital product. It is designed to complement EARL and ACT Rules, not to replace them.

2. Use Cases

  • Developers and agencies documenting the accessibility status of their themes, templates, apps, and plugins.
  • Website operators and project managers evaluating whether a product meets their accessibility requirements.
  • Web editors writing accessibility statements without performing full WCAG audits.
  • Accessibility testers publishing results in a structured, machine-readable way.
  • CMS plugins automatically generating JSON-LD, accessibility.txt, or accessibility statements.

3. Relationship to EARL and ACT Rules

This specification complements W3C technologies:

3.1 ACT Rules

ACT Rules define how tests should be performed. They do not define report formats or project metadata. Use ACT Rules to standardize the testing process.

3.2 EARL

EARL defines a framework for expressing individual test assertions in RDF. It does not define conformance summaries, project details, or accessibility statements. Use EARL to exchange raw test results.

This document defines the project-level container for results derived from ACT tests, EARL assertions, manual evaluation, or hybrid audits.

4. The Accessibility Metadata Format

The format is expressed as JSON and formally defined in a JSON Schema. The top-level structure contains:

  • specVersion – Format version
  • language – Language of the report
  • project – Metadata about the evaluated product
  • evaluation – Details of the accessibility assessment

A complete JSON Schema and example files are available in the corresponding repository.

5. Example (accessibility.json)

{
  "specVersion": "1.0.0",
  "language": "en",
  "project": {
    "name": "Example Theme",
    "version": "1.0.0",
    "contact": {
      "email": "accessibility@example.org"
    },
    "scope": {
      "coverage": "templates",
      "legalBasis": ["WCAG 2.2"]
    }
  },
  "evaluation": {
    "standard": "WCAG",
    "version": "2.2",
    "conformance": "partially_conformant",
    "issues": [
      {
        "criterion": "1.4.3",
        "reason": "Low text contrast in footer links."
      }
    ]
  }
}

6. Generating JSON-LD for Schema.org

The metadata in accessibility.json can be used to automatically generate a Schema.org WebSite JSON-LD block using accessibilitySummary. 

{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "name": "Example Theme Demo",
  "url": "https://example.org/",
  "inLanguage": "en",
  "accessibilitySummary": "This website is partially conformant with WCAG 2.2..."
}

See the full documentation on JSON-LD integration for details.

7. Generating accessibility.txt

The format can also be used to generate a lightweight, human-readable accessibility.txt file as proposed by Bogdan Cerovac (2023). 

Contact: mailto:accessibility@example.org
Preferred-Languages: en, de
WCAG-Conformance: partially conformant
Last-Update: 2025-11-25
Generated-From: accessibility.json

9. Conformance

This specification uses the keywords MUST, SHOULD, and MAY as defined in RFC 2119. JSON documents conform when they match the normative JSON Schema.
